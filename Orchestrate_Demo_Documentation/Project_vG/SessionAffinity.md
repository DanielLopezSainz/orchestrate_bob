# Session Affinity & Conversational Memory (Thread ID Sync)

## 1. The Problem: "Agent Amnesia"
In a Single Page Application (SPA) like React, local state is volatile. When a user refreshes the page, or when an OpenID Connect (OIDC) authentication flow redirects the browser, all local React variables are wiped out. 

Watsonx Orchestrate remembers conversations by associating them with a unique "Thread ID" (e.g., `X-IBM-THREAD-ID`). If the frontend loses this ID and sends a blank request, Watsonx assumes it is a brand-new user, resulting in the agent forgetting the context of the conversation.

## 2. Why Previous Iterations Failed
Initially, we attempted to extract the Thread ID from the HTTP Response Headers returned by the Node.js backend. 

However, because we use Server-Sent Events (SSE) for a real-time typing effect (`stream: true`), the HTTP headers must be flushed and sent to the browser *before* Watsonx finishes generating the Thread ID. Therefore, the header was always empty.

## 3. The Solution: Bi-Directional State Sync
To solve this reliably, we implemented a Bi-Directional Sync between the React Frontend and the Express Backend's Session layer. 

The Backend acts as the "Single Source of Truth" (vault), while the Frontend acts as the "Active Reader" (extractor).

Here is the exact step-by-step flow and the code that makes it happen:

---

### Step 1: The Initial Request
* The user sends their very first message.
* React's `useChat.js` starts with a blank slate (`threadId: null`).
* It sends a POST request to `/api/chat`.

**Frontend Code: `useChat.js`**
```javascript
const [threadId, setThreadId] = useState(null); // Starts empty

const sendMessage = async (text) => {
    // ...
    const response = await fetch('/api/chat', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ message: text, threadId }), // Sends null initially
    });
    // ...
}
```

---

### Step 2: The Payload Extraction (Frontend)
* Watsonx creates a new conversation thread and begins streaming the response.
* Because it's a stream, `useChat.js` uses a string buffer to reassemble chunks.
* Deep inside one of these JSON chunks, Watsonx includes the newly generated `thread_id`. `useChat.js` detects it and saves it to React's local state.

**Frontend Code: `useChat.js`**
```javascript
// Inside the SSE stream reader loop...
try {
    const data = JSON.parse(dataStr);
    
    // Extracted directly from the JSON body, NOT the headers
    if (data.thread_id) {
        setThreadId(prevThreadId => {
            // Update React's memory so the next message uses this ID
            return data.thread_id; 
        });
    }
    // ...
} catch (err) {
    // Buffer handles split chunks silently
}
```

---

### Step 3: The Backend Vault Sync
* The user sends their second message. React now sends the extracted `threadId`.
* The Node.js `server.js` route intercepts this request. 
* It saves this ID permanently into the user's secure `express-session` cookie.
* It then forwards the request to Watsonx, appending the header.

**Backend Code: `server.js`**
```javascript
app.post('/api/chat', protect, async (req, res) => {
    let activeThreadId = req.body.threadId;

    if (activeThreadId) {
        // The frontend sent an ID! Save it to the secure session vault.
        req.session.threadId = activeThreadId;
        req.session.save();
    }
    
    // Forward to Watsonx Client...
    const response = await streamChat(req.body.message, activeThreadId);
    // ...
});
```

---

### Step 4: The "Amnesia" Recovery (Page Refresh)
* The user refreshes the browser. React boots up blank (`threadId: null`).
* The user types a message. React sends `threadId: null` to the backend.
* The Node.js backend intercepts the request and checks its session vault. 
* It silently overrides the blank request, injects the saved Thread ID, and forwards it to Watsonx, perfectly repairing the broken context.

**Backend Code: `server.js`**
```javascript
app.post('/api/chat', protect, async (req, res) => {
    let activeThreadId = req.body.threadId; // This is null right now

    if (activeThreadId) {
        req.session.threadId = activeThreadId;
        req.session.save();
    } else if (req.session.threadId) {
        // AMNESIA RECOVERY: The frontend forgot, but the backend remembers.
        // Override the null value with the saved session value.
        activeThreadId = req.session.threadId; 
    }
    
    // Forward to Watsonx Client with the recovered ID...
    const response = await streamChat(req.body.message, activeThreadId);
    // ...
});
```

---

## Summary of Architectural Responsibilities
* **Frontend (`useChat.js`)**: Responsible for buffering the SSE stream, parsing the JSON, and extracting the `thread_id` dynamically generated by Watsonx.
* **Backend (`server.js`)**: Responsible for maintaining the `express-session` cookie, acting as a fail-safe vault to protect the Thread ID across page loads.
